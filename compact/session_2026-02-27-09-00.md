## Session Summary

This session focused on three major UX improvements to the interactive REPL:

1. **Event-based streaming architecture**: Changed from per-delta `text` events to buffered `text_complete` + `tool_use` events. This prevents incomplete markdown from being rendered mid-stream and allows proper detection of tool calls.

2. **Keypress hint system**: Implemented (and iterated through multiple approaches) an inline ghost hint system for slash commands. The final version renders hints on the same line as the prompt, truncated to terminal width, and only for ASCII-safe slash commands (avoiding CJK cursor corruption issues).

3. **Terminal cleanup**: Added full terminal clear (including scrollback buffer) on REPL startup for a clean slate.

The session involved significant trial-and-error with terminal escape sequences, particularly around cursor positioning for keypress hints. Multiple bugs were encountered and fixed related to terminal scrolling, CJK text width calculation, and cursor synchronization.

**Attempt 3: Inline hints** (`58f7c73`)

- Used `\x1b[{col}G` to move cursor to absolute column
- **Problem**: Works for ASCII but breaks CJK (Korean, Japanese) input
  - Issue: `line.length` ≠ display width for multi-byte characters
  - Cursor position calculation wrong for CJK, corrupts terminal
- **Result**: Partially working, but breaks Korean input

**Attempt 4: Truncated inline hints** (`7e1c2af`)
- Same approach but truncate hint to fit terminal width
- Added `\u2026` (ellipsis) when truncated
- **Problem**: Still breaks CJK input due to length vs width mismatch
- **Result**: Works for English, breaks Korean

**Attempt 5: Slash-command-only hints** (`de93406`)
- **Key insight**: Only render escape codes for slash commands (ASCII-safe)
- Early return for non-slash input, let readline handle it natively
- For slash commands: clear from end of input to end of line, write hint, restore cursor
- **Solution**:
  ```typescript
  // Only touch terminal for / commands (ASCII-only)
  if (!line.startsWith("/")) return;

  const endCol = PROMPT_VIS + line.length + 1;
  const cursorCol = PROMPT_VIS + rl.cursor + 1;

  // Clear previous hint, write new hint, restore cursor
  process.stdout.write(`\x1b[${endCol}G\x1b[K`);
  process.stdout.write(`  \x1b[2m${hint}\x1b[0m`);
  process.stdout.write(`\x1b[${cursorCol}G`);
  ```
- **Result**: Works perfectly for slash commands, Korean input unaffected

## Decisions Made

### Decision 24: Buffered Text Events (not Per-Delta)
- **Date**: 2026-02-27 ~09:00
- **Context**: Streaming text emitted per-delta causes incomplete markdown rendering
- **Options**:
  A. Emit per-delta `text` events (original)
  B. Buffer deltas, emit on content_block_stop as `text_complete`
  C. Delay emission by fixed time (e.g., 100ms debounce)
- **Choice**: B (buffer + flush on block stop)
- **Rationale**:
  - Markdown patterns (`**bold**`) need complete strings to parse correctly
  - Tool use events need to be distinguished from text
  - Content block boundaries are natural flush points
  - No artificial delays (option C would add latency)
  - Handles both streaming and complete message formats
- **Impact**: Changed event architecture in streamer.ts, updated all listeners in repl.ts
- **Trade-offs**: Slight buffering delay, but imperceptible to users (< 50ms typically)
- **Reversible**: Yes (can switch back to per-delta if needed)

### Decision 25: Tool Use Events Separate from Text
- **Date**: 2026-02-27 ~09:00
- **Context**: Tool calls (Read, Bash, etc.) were not visible to user during execution
- **Options**:
  A. Include tool calls in text output (as JSON)
  B. Emit separate `tool_use` events
  C. Show tool calls only in final summary
- **Choice**: B (separate events)
- **Rationale**:
  - Users should see what tools agent is using (transparency)
  - Tool calls are actions, not text — semantically different
  - Allows custom rendering (yellow `▸` prefix, concise format)
  - Can extract key parameter (file_path, command, pattern) for brevity
- **Impact**: Added `ToolUseEvent` interface, `toolUse()` renderer function
- **Trade-offs**: More event types to handle, but cleaner separation
- **Reversible**: Yes (can merge back into text stream)

### Decision 26: Keypress Hints Only for Slash Commands
- **Date**: 2026-02-27 ~09:15
- **Context**: Inline hints break CJK input due to string length vs display width mismatch
- **Options**:
  A. Calculate exact display width for all input (using wcwidth library)
  B. Disable hints entirely
  C. Only show hints for slash commands (ASCII-safe)
- **Choice**: C (slash-command-only)
- **Rationale**:
  - Slash commands are always ASCII (safe for `.length` calculations)
  - Natural language input often contains CJK (user is Korean)
  - wcwidth library adds dependency + complexity for marginal benefit
  - Early return for non-slash input has zero overhead
  - User mainly uses hints for discovering slash commands anyway
- **Impact**: Added `if (!line.startsWith("/")) return;` guard in keypress handler
- **Trade-offs**: No hints for natural language input, but that's not the primary use case
- **Reversible**: Yes (can add wcwidth calculation later if needed)

### Decision 27: Terminal Clear on REPL Startup
- **Date**: 2026-02-27 ~09:30
- **Context**: Previous terminal output visible when starting REPL
- **Options**:
  A. Leave terminal as-is (user can clear manually)
  B. Clear visible screen only (`\x1b[2J`)
  C. Clear screen + scrollback (`\x1b[2J\x1b[3J`)
- **Choice**: C (clear all)
- **Rationale**:
  - Clean slate improves focus (no distractions from previous commands)
  - Scrollback clear prevents accidental scroll-up confusion

## Errors Encountered & Resolved

### Error 11: Keypress Hints Cause Terminal Scroll Corruption
- **When**: Phase 2, Attempt 1 (`bb6e7d0`)
- **Symptom**: Prompt jumps around screen, text overlaps, cursor position wrong
- **Root Cause**: Used `\x1b[s\n\x1b[2K\x1b[u` (save cursor, newline, clear, restore)
  - `\n` causes terminal scroll when cursor is at bottom of screen
  - Scroll happens BEFORE cursor position is saved, making restore position wrong
  - Terminal thinks cursor is 1 line higher than it actually is
- **Solution**: Removed newline approach, switched to inline rendering
- **Files Affected**: `src/repl/repl.ts`
- **Prevention**: Never use `\n` in escape sequences during readline input
- **Commit**: `1a05897` (fix: render keypress hints inline instead of below prompt line)

### Error 12: Relative Cursor Movement Still Breaks
- **When**: Phase 2, Attempt 2 (`1a05897`)
- **Symptom**: Same scroll corruption as Error 11
- **Root Cause**: Still used `\n` followed by `\x1b[A` to move cursor back up
  - Terminal scrolls on `\n`, THEN moves cursor up
  - Scroll invalidates all subsequent cursor positions
  - Fundamentally incompatible with readline's line tracking
- **Solution**: Remove `\n` entirely, render on same line
- **Files Affected**: `src/repl/repl.ts`
- **Prevention**: Inline rendering only, no multi-line escape sequences
- **Commit**: `58f7c73` (fix: remove keypress hint system that corrupts terminal output)

### Error 13: CJK Input Corrupts Cursor Position
- **When**: Phase 2, Attempt 3 (`58f7c73`)
- **Symptom**: Korean input causes cursor to jump, characters appear in wrong places
- **Root Cause**: Used `line.length` to calculate cursor position
  - Korean characters are multi-byte UTF-8 (3 bytes per character)
  - String `.length` counts JavaScript string units (UTF-16 code units)
  - Display width ≠ string length for multi-byte characters
  - Example: "한글" has length 2 but may display differently depending on font
  - Escape sequence `\x1b[{col}G` uses display column, not string length
- **Solution**: Only apply escape codes for slash commands (ASCII-safe)
  - Early return for non-slash input
  - Slash commands are always ASCII (`/help`, `/lang en`, etc.)
  - Let readline handle CJK input natively without interference
- **Files Affected**: `src/repl/repl.ts`
- **Prevention**:
  - Never assume string length = display width for multi-byte text
  - Use wcwidth library if exact calculation needed, OR
  - Constrain escape sequences to ASCII-only contexts
- **Commit**: `de93406` (fix: skip escape codes for non-slash input to prevent CJK cursor corruption)

## Technical Insights Preserved

### Streaming Event Architecture

**Pattern**: Content block lifecycle for buffered events
```typescript
// State
private textBuffer = "";
private currentBlockType: string | null = null;

// content_block_start: detect type, emit tool_use immediately
if (msg.type === "content_block_start") {
  this.currentBlockType = msg.content_block?.type ?? null;
  if (this.currentBlockType === "tool_use") {
    this.emit("tool_use", { name, id, input });
  } else {
    this.textBuffer = ""; // reset for text block
  }
}

// content_block_delta: accumulate in buffer
if (msg.type === "content_block_delta") {
  if (msg.delta?.text) {
    this.textBuffer += msg.delta.text;
  }
}

// content_block_stop: flush buffer
if (msg.type === "content_block_stop") {
  if (this.currentBlockType === "text" && this.textBuffer) {
    this.emit("text_complete", this.textBuffer);
    this.textBuffer = "";
  }
  this.currentBlockType = null;
}
```

**Why this pattern**:
- Markdown patterns need complete strings (`**bold**` won't parse if split mid-pattern)
- Tool use needs immediate visibility (user wants to know what's happening NOW)
- Natural flush points at block boundaries
- Handles both streaming (block_start/delta/stop) and complete (assistant message) formats

### Terminal Escape Sequence Rules for Readline

**Safe operations**:
- `\x1b[{col}G` — move cursor to absolute column (IF column calculation is correct)
- `\x1b[K` — clear from cursor to end of line
- `\x1b[2m` / `\x1b[0m` — dim text / reset formatting
- `\r` — carriage return to start of line

**Unsafe operations**:
- `\n` — NEVER use during readline input (causes scroll at bottom of screen)
- `\x1b[A` / `\x1b[B` — avoid vertical movement (readline owns line tracking)
- `\x1b[s` / `\x1b[u` — save/restore cursor (unreliable with readline)

**Safe pattern for inline hints**:
```typescript
const endCol = PROMPT_VIS + line.length + 1; // where user input ends
const cursorCol = PROMPT_VIS + rl.cursor + 1; // where cursor currently is

// 1. Move to end of input, clear to end of line
process.stdout.write(`\x1b[${endCol}G\x1b[K`);

// 2. Write hint text (dim)
process.stdout.write(`  \x1b[2m${hint}\x1b[0m`);

// 3. Restore cursor to typing position
process.stdout.write(`\x1b[${cursorCol}G`);
```

**IMPORTANT**: This only works if `line.length` equals display width
- Safe: ASCII input (`/help`, `/lang en`)
- Unsafe: CJK input (`한글 입력`)
- Solution: Only apply to slash commands (ASCII-safe)

### String Length vs Display Width

**The problem**:
- JavaScript string `.length` = number of UTF-16 code units
- Terminal display width = number of terminal columns occupied
- Multi-byte characters: length ≠ width

**Examples**:
```typescript
"hello".length === 5         // display width: 5 ✓ match
"/help".length === 5         // display width: 5 ✓ match
"한글".length === 2           // display width: 4 ✗ mismatch
"こんにちは".length === 5     // display width: 10 ✗ mismatch
```

**Solutions**:
1. **Use wcwidth library**: Calculate exact display width
   - Pros: Accurate for all characters
   - Cons: Extra dependency, complexity
2. **Constrain to ASCII contexts**: Only where input is guaranteed ASCII
   - Pros: Zero overhead, no dependencies
   - Cons: Limited to ASCII use cases
3. **Avoid the problem**: Don't use escape codes that need column calculations
   - Pros: Safest, no edge cases
   - Cons: Can't implement certain features

**Decision**: Used solution 2 (ASCII-only contexts) for slash commands
