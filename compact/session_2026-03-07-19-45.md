## Decisions Made

### Decision 1: Reference-First Protocol as [PRIORITY 0]
- **Context**: Design agent was producing generic "AI-looking" designs when no references provided
- **Options considered**:
  - A) Add more design rules and constraints
  - B) Provide better examples in profile
  - C) Make references MANDATORY via 5-step protocol
- **Choice**: C (Reference-First Protocol)
- **Rationale**: Rules can be ignored, examples can be misunderstood, but a mandatory protocol with verification steps forces the agent to think like a real product designer
- **Impact**: Every UI generation now requires declared references, making "AI look" impossible
- **Reversible**: No (this is a fundamental design philosophy)

### Decision 2: Extract Real Design Tokens (not generic guidelines)
- **Context**: Needed concrete reference data for agents to follow
- **Options considered**:
  - A) Describe design styles in prose (e.g., "Linear uses clean, minimal design")
  - B) Extract actual hex values, font stacks, spacing systems
- **Choice**: B (Real extracted tokens)
- **Rationale**: Prose is ambiguous. `#3182f6` is not. Exact values force precision.
- **Impact**: Reference database is now a constraint system, not a suggestion list
- **Reversible**: No (exact values are the entire point)

### Decision 3: Anonymize All Service Names
- **Context**: User concerned about legal implications of naming real products
- **Options considered**:
  - A) Keep real names (Toss, Linear, etc.)
  - B) Use generic descriptions ("Korean fintech", "PM tool")
  - C) Use codes (KR-1, GL-1, etc.)
- **Choice**: C (Anonymous codes)
- **Rationale**: Generic descriptions lose specificity, codes preserve reference structure while removing legal risk
- **Impact**: All references now use KR-/GL- codes, no URLs, no proprietary font names
- **Reversible**: Yes (could add mapping table, but not recommended)

### Decision 4: Standardize Korean Fonts to Pretendard Only
- **Context**: Profile had inconsistent Korean font recommendations (Pretendard, Noto Sans KR, Noto Serif KR)
- **Options considered**:
  - A) Keep multiple options for variety
  - B) Standardize to Pretendard only
  - C) Use system fonts only
- **Choice**: B (Pretendard only)
- **Rationale**: Modern Korean SaaS has converged on Pretendard (200-900 weights, excellent rendering, wide adoption). Multiple fonts create decision paralysis.
- **Impact**: All Korean text uses Pretendard → system fallback
- **Reversible**: Yes (could add Noto back if needed)

### Decision 5: Use AG Grid for CRM Demo (not raw HTML table)
- **Context**: Initial CRM demo used raw HTML table, looked "too AI-like"
- **Options considered**:
  - A) Style raw HTML table better
  - B) Use production-grade data grid library (AG Grid)
- **Choice**: B (AG Grid Community)
- **Rationale**: Real CRM products don't use styled HTML tables. They use libraries like AG Grid. Using the same tools produces the same look.
- **Impact**: CRM demo now uses AG Grid with Relate-style dark theme customization
- **Reversible**: Yes (but would lose production realism)

### Decision 6: Include "Does NOT use" Anti-Patterns
- **Context**: References need to show what to avoid, not just what to include
- **Options considered**:
  - A) Only show positive patterns (what to do)
  - B) Also show negative patterns (what NOT to do)
- **Choice**: B (Include anti-patterns)
- **Rationale**: "Linear does NOT use shadows, dark sidebar, or bento grids" is as important as "Linear uses compact nav". Constraints by subtraction.
- **Impact**: Each reference now includes "Does NOT use" list
- **Reversible**: No (critical for preventing AI defaults)

### Decision 7: Make Protocol 5 Steps (not 3 or 7)
- **Context**: Needed structured process for reference-based design
- **Options considered**:
  - A) 3 steps (identify, generate, verify)
  - B) 5 steps (identify, extract, declare, generate, verify)
  - C) 7+ steps (add research, iteration, etc.)
- **Choice**: B (5 steps)
- **Rationale**: 3 is too loose (no explicit declaration), 7+ is too rigid (slows execution). 5 is the minimum viable constraint.
- **Impact**: Every UI generation follows IDENTIFY → EXTRACT → DECLARE → GENERATE → VERIFY
- **Reversible**: No (this is the core protocol)
