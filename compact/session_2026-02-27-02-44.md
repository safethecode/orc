## Decisions Made

### Decision 1: Binary Command Name
- **Context**: Need CLI command name for the orchestrator
- **Options considered**: `orch`, `orc`, `orchestrator`
- **Choice**: `orc`
- **Rationale**: Shorter, easier to type, distinctive
- **Impact**: package.json bin entry, all documentation
- **Reversible**: Yes (just rename bin entry)

### Decision 2: Runtime Choice - Bun
- **Context**: TypeScript runtime for the application
- **Options considered**: Node.js, Bun, Deno
- **Choice**: Bun
- **Rationale**:
  - Native TypeScript support (no transpilation)
  - Built-in SQLite (bun:sqlite)
  - Faster than Node
  - Great developer experience
- **Impact**: All tooling, package.json scripts
- **Reversible**: Partially (would need to rewrite DB layer)

### Decision 3: tmux for Session Isolation
- **Context**: Need to run multiple AI agents concurrently without interference
- **Options considered**: Docker containers, screen, tmux, native processes
- **Choice**: tmux
- **Rationale**:
  - Lightweight (no container overhead)
  - Easy to inspect sessions (`tmux attach`)
  - Standard on Unix systems
  - Good terminal multiplexing
- **Impact**: Requires tmux installation, session/terminal.ts implementation
- **Reversible**: Yes (could swap for screen or containers)

### Decision 4: SQLite for State Management
- **Context**: Need persistent storage for agents, tasks, messages
- **Options considered**: PostgreSQL, SQLite, JSON files, in-memory
- **Choice**: SQLite with WAL mode
- **Rationale**:
  - No external dependencies
  - Built into Bun runtime
  - WAL mode for good concurrency
  - File-based (easy backup/restore)
- **Impact**: db/schema.ts, db/store.ts
- **Reversible**: Partially (would need migration scripts)

### Decision 5: Ink for TUI Dashboard
- **Context**: Need visual dashboard to monitor agents
- **Options considered**: ncurses, blessed, ink, raw terminal codes
- **Choice**: Ink (React for CLI)
- **Rationale**:
  - Component-based (familiar React patterns)
  - Good terminal rendering
  - Easy state management with hooks
  - Active maintenance
- **Impact**: tui/*.tsx files, React dependency
- **Reversible**: Yes (could rewrite with different framework)

### Decision 6: MCP for Agent Communication Protocol
- **Context**: Need standard protocol for agents to communicate with orchestrator
- **Options considered**: Custom protocol, MCP, gRPC, REST API
- **Choice**: MCP (Model Context Protocol)
- **Rationale**:
  - Designed for AI agent communication
  - Standard protocol from Anthropic
  - stdio transport (simple)
  - Tool-based interface
- **Impact**: messaging/mcp-server.ts, @modelcontextprotocol/sdk dependency
- **Reversible**: Yes (protocol is abstracted)

### Decision 7: File Ownership for Conflict Prevention

- **Options considered**: File locks, ownership declaration, no prevention

- **Rationale**:

## Errors Encountered & Resolved

### Error 1: Binary name mismatch
- **When**: Phase 1 (commit 5)
- **Symptom**: package.json had `"orch": "./src/index.ts"` but user wanted `orc`
- **Root cause**: Initial implementation used longer name
- **Solution**: Changed bin entry to `"orc": "./src/index.ts"`
- **Files affected**: package.json
- **Prevention**: Confirm naming preferences upfront

### Error 2: TypeScript type errors with unknown[]
- **When**: Phase 2-8 (commits 42, 44)
- **Symptom**: TypeScript compiler errors: `Type 'unknown[]' is not assignable to type 'string[]'`
- **Root cause**: SQLite prepared statement bindings typed as unknown[]
- **Solution**: Changed to typed arrays:
  - `[agentId] as [string]`
  - `[status, agentId] as [string, string]`
  - `[taskId, agentId] as [number, string]`
- **Files affected**:
  - src/db/store.ts (multiple methods)
  - src/messaging/inbox.ts (getMessages query)
- **Prevention**: Use typed arrays from the start, enable strict mode

### Error 3: Deep merge type incompatibility
- **When**: Phase 2-8 (commit 43)
- **Symptom**: Type error in config/loader.ts: `deepMerge<OrchestratorConfig>(...)` type mismatch
- **Root cause**: Generic type inference issues with deep merge function
- **Solution**: Explicit type assertion in deepMerge call
- **Files affected**: src/config/loader.ts
- **Prevention**: Better type definitions for utility functions

### Error 4: allowImportingTsExtensions not set
- **When**: Phase 2-8 (commit 41)
- **Symptom**: TypeScript errors about importing .ts files
- **Root cause**: Bun requires this flag for .ts imports
- **Solution**: Added `"allowImportingTsExtensions": true` to tsconfig.json
- **Files affected**: tsconfig.json
- **Prevention**: Use Bun-specific tsconfig template

### Error 5: Missing executable permission
- **When**: Phase 2-8 (commit 45)
- **Symptom**: `orc` command not executable
- **Root cause**: src/index.ts didn't have executable permission
- **Solution**: `chmod +x src/index.ts` and added shebang `#!/usr/bin/env bun`
- **Files affected**: src/index.ts
- **Prevention**: Set permissions when creating CLI entry points

### Error 6: tsc not installed
- **When**: Current session (compact history phase)
- **Symptom**: `bun run typecheck` fails with "tsc not found"
- **Root cause**: TypeScript not in devDependencies
- **Solution**: Not yet resolved (Bun's built-in type checker works)
- **Files affected**: None yet
- **Prevention**: Add typescript to devDependencies if tsc is needed

## Open Questions

1. **REPL interface choice**: readline vs Ink vs custom?
   - readline: Simple, text-based, standard
   - Ink: Rich UI, can show agent status in panels
   - Custom: Most control, most work

2. **Natural language parsing**: Local LLM vs API vs keyword-based?
   - Local: Fast, private, but needs model
   - API: Accurate, but latency and cost
   - Keyword: Simple, but limited

3. **Output streaming format**: Plain text vs structured vs mixed?
   - Plain: Easy to read, but hard to parse
   - Structured: Easy to parse, but verbose
   - Mixed: Best of both, but complex

4. **Session state storage**: In-memory vs SQLite vs Redis?
   - In-memory: Fast, but lost on restart
   - SQLite: Persistent, but I/O overhead
   - Redis: Fast + persistent, but external dependency

5. **Error handling in auto-spawn**: Retry vs fail vs ask user?
   - Retry: Resilient, but might waste resources
   - Fail: Simple, but poor UX
   - Ask user: Best UX, but breaks flow

6. **TypeScript compiler**: Install tsc or rely on Bun?
   - Install tsc: Standard, better tooling
   - Bun only: Simpler, but less ecosystem support
